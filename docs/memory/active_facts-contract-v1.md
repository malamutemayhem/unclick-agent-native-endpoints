# Active Facts Contract V1

**Status**: Draft  
**Last updated**: 2026-04-29  
**Owner**: `🦾`  
**Purpose**: Pre-work contract draft for Cluster A before Chris's design call

## Why this exists

The current `active_facts` behavior has drifted into two problems:

1. startup context can surface operational self-report noise instead of durable user knowledge
2. invalidation, ranking, and access-metric behavior are not fully aligned across managed, BYOD, and local paths

The room already has four related lanes behind this:

- `4a810c0a`
- `0bc231c4`
- `20b991ab`
- `6a3bb792`

This document turns those scattered concerns into one draft contract so the eventual design call is about decisions, not about rediscovering the problem.

## TODO closure map

This V1 contract is intended to close these four lanes with one shared ruleset:

- `4a810c0a`: write-time eligibility must distinguish durable user knowledge from operational self-report
- `0bc231c4`: read-time filtering must keep invalidated and operational-noise rows out of the surfaced startup set
- `20b991ab`: startup-context metric bumps must apply to surfaced facts, not merely to a broader eligible pool
- `6a3bb792`: managed, BYOD, and local implementations must produce materially equivalent `active_facts` behavior

## Scope

This contract covers only the `active_facts` portion of startup context.

It does not try to redesign:

- business context loading
- recent session summaries
- library index loading
- semantic search ranking generally
- fact extraction broadly

## Product goal

`active_facts` should help a fresh agent act like it already knows the user.

That means the set should prefer:

- durable user facts
- standing operationally relevant facts
- high-signal recent facts

It should avoid:

- repetitive cron self-reports
- transient system narration
- invalidated facts
- low-value operational chatter that crowds out durable context

## Contract shape

The contract has three layers:

1. **write-time eligibility metadata**
2. **read-time selection and filtering**
3. **post-read metric behavior**

All three need to be explicit or the behavior will drift again by deployment mode.

## Definitions

### Fact

A row in `mc_extracted_facts` or the equivalent local/BYOD store.

### Active fact

A fact eligible to appear in startup context for immediate session grounding.

### Operational self-report

A fact generated by the system or workflow machinery to narrate internal state, recurring heartbeat activity, or background maintenance rather than to preserve durable user knowledge.

### Durable user fact

A fact whose likely value survives beyond the immediate operational moment and improves a future agent's decision quality.

### Startup-eligible fact

A fact that is still valid and whose write-time metadata does not disqualify it from consideration for `active_facts`.

### Surfaced set

The final returned `active_facts` array after all filtering, ranking, and top-N trimming are complete.

### Eligible pool

The larger set of still-valid candidate facts considered before the final surfaced set is chosen.

## Write-time contract

### W1. Every fact must carry enough provenance for startup-context filtering

The current room evidence shows `source_type` alone is too blunt because ordinary `addFact` writes already stamp `manual`.

So V1 should not rely on a binary `manual` versus non-`manual` split.

Instead, every write path that can create a fact MUST persist enough metadata to let startup-context determine whether the fact is:

- durable user knowledge
- operational self-report
- otherwise excluded from startup eligibility

The mechanism may be any of:

- a narrower source marker for cron/system self-reports
- a category or subtype reserved for operational narration
- a startup-eligibility flag

The key rule is:

- startup-context filtering must be able to separate durable user knowledge from operational self-report without suppressing ordinary user-created facts

Rows that lack the minimum provenance to make that distinction MUST NOT be treated as implicitly durable user facts.

### W2. Cron and system self-reports must not masquerade as ordinary user facts

If a job writes heartbeat-style or internal-status facts, those rows need a distinct marker at write time.

Examples of rows that should be markable as non-primary startup candidates:

- repeated heartbeat summaries
- operational lane narration
- recurring internal maintenance observations

V1 does not require deleting these rows.
It requires writing them in a way that startup-context can reason about them safely.

More crisply:

- operational narration MAY be stored
- operational narration MUST be marked at write time
- unmarked operational narration is a contract violation, not a ranking quirk

### W3. Invalidation is stronger than relevance

Once a fact is invalidated, it is no longer eligible for `active_facts`, regardless of recency, confidence, or prior hotness.

This needs to hold in:

- managed SQL path
- BYOD SQL path
- local file-backed path(s)

Invalidation MUST be treated as a hard exclusion at read time, not as a soft ranking signal.

### W4. Eligibility metadata should not overfit one current bug

The marker chosen for cron pollution should generalize to future low-value operational narration too.

The contract should avoid baking in one hyper-specific string like "heartbeat" as the only special case.

## Read-time contract

### R1. `active_facts` is a curated top set, not a dump of all hot facts

The startup set should be intentionally scarce.

Its job is not to maximize recall.
Its job is to maximize immediate usefulness for a fresh session.

That means implementations MAY examine a broader eligible pool, but the product contract is defined by the surfaced set, not by the pre-trim candidate list.

### R2. Invalidated facts are always excluded

No invalidated fact may appear in `active_facts`.

This is the simplest non-negotiable rule in the contract.

If a fact has any invalidation state recognized by that deployment mode, it MUST be removed before ranking for the surfaced set.

### R3. Operational self-reports are excluded or strongly demoted

Rows marked as cron/system operational narration should not occupy meaningful startup slots unless explicitly justified by future policy.

The default V1 posture should be:

- exclude from `active_facts`, or
- place behind a very strong demotion rule that keeps them out of the visible top set under normal conditions

The room evidence suggests exclusion is the clearer default.

V1 therefore sets the default rule as:

- marked operational self-report MUST NOT appear in the surfaced set

An implementation may keep such rows in storage or in an internal candidate pool, but it MUST NOT let them consume visible startup slots under ordinary V1 behavior.

### R4. Durable user facts win over operational recency

Recency matters, but not enough to let low-value self-report noise displace durable high-signal user facts.

Ranking policy should prefer:

1. still-valid durable facts
2. high-confidence user-relevant recent facts
3. only then any operational or system narration if ever allowed at all

If future policy introduces a narrow allowlist for operational rows, that policy MUST be explicit and MUST remain subordinate to still-valid durable user facts.

### R5. The payload contract stays minimal unless intentionally expanded

Today `active_facts` payloads are stripped to:

- `fact`
- `category`
- `confidence`
- `created_at`

That means client-side filtering cannot solve this problem.

So V1 assumes filtering and ranking happen server-side unless the contract is intentionally expanded later.

### R6. Behavior must be parity-locked across all startup-context paths

The same effective `active_facts` rules must apply to:

- managed cloud SQL
- BYOD SQL/schema path
- local mode in `packages/mcp-server`
- local mode in `packages/memory-mcp`

If one path drifts, the product story becomes deployment-dependent again.

Parity here means:

- the same kinds of rows are startup-eligible
- the same kinds of rows are excluded
- invalidation has the same force
- metric bumps reflect the same surfaced-set rule even if implementation details differ

## Metric-update contract

### M1. Only surfaced `active_facts` should receive startup-context access credit

Today the room found a cross-mode drift:

- managed and BYOD update the broader eligible hot pool after selecting top facts
- local mode currently does not bump active-fact access metrics at all

V1 should lock one rule:

- startup-context access metrics apply only to the facts actually surfaced in the returned `active_facts` set

This best matches the product meaning of "this fact was actually loaded into context."

Selecting from an eligible pool is not enough.
A fact MUST be in the surfaced set before startup-context can bump `access_count`, `last_accessed`, or equivalent hotness fields.

### M2. Hidden or excluded facts must not get hotness reinforcement

If a fact is filtered out because it is invalidated or because it is operational self-report noise, startup-context reads must not keep refreshing its `access_count` or `last_accessed`.

Otherwise the ranking system silently reinforces the very rows it is trying not to surface.

This distinction is required:

- being considered during read-time selection does not earn a metric bump
- being returned in the surfaced set does earn a metric bump

The contract therefore separates:

- eligible pool membership
- surfaced set membership
- metric reinforcement

Only the third follows from the second, never from the first alone.

### M3. Local parity matters too

If local mode keeps metric-free behavior for practical reasons, that exception should be explicit.

But the preferred V1 contract is parity:

- same surfaced-set semantics everywhere

If any local path cannot bump metrics yet, it MUST still:

- return the same surfaced set it would have returned under parity
- avoid bumping hidden facts
- document the temporary gap as an implementation limitation rather than a different contract

## Recommended V1 rule summary

The smallest durable contract looks like this:

1. mark cron/system self-report rows distinctly at write time
2. exclude invalidated rows from `active_facts`
3. exclude or strongly demote marked operational narration from `active_facts`
4. rank durable user facts above operational recency
5. update access metrics only for the surfaced top set, never for merely eligible hidden rows
6. enforce the same rules across managed, BYOD, and local paths

## Acceptance criteria

### A1. Pollution regression

Given a mix of durable user facts and repeated cron self-report facts, startup context should not surface the cron self-report rows in the visible top set under ordinary conditions.

### A2. Invalidation parity

Given an invalidated formerly-hot fact, repeated startup-context calls must:

- not surface it in `active_facts`
- not refresh its startup-context access metrics
- not allow it back into the surfaced set unless the invalidation is explicitly cleared by product-supported behavior

### A3. Cross-mode parity

The same fixture set should produce materially equivalent `active_facts` behavior in:

- managed
- BYOD
- local

### A4. Metric clarity

Startup-context reads should only reinforce facts that were actually returned in the startup payload.

Facts that were considered but trimmed, excluded, or invalidated should show no startup-context reinforcement from that read.

## Test recommendation

The cheapest durable regression seam is a dedicated startup-context test that:

1. seeds one durable user fact
2. seeds one or more cron self-report facts
3. optionally seeds one invalidated hot fact
4. calls startup context
5. asserts the returned `active_facts` set favors the durable fact and excludes the bad candidates

This is better than relying on live observation alone.

## Out of scope

This V1 contract does not decide:

- the exact field name for the cron/system marker
- a fully general memory ranking formula for every retrieval path
- whether startup-context should eventually expose more provenance in payloads
- whether operational rows should be deleted, archived, or merely demoted

Those are implementation or later-phase questions.

## Final lock

`active_facts` is not "all hot facts."

It is a curated startup-context surface for the most useful still-valid knowledge a fresh agent should load first.

That means:

- invalidated rows stay out
- operational self-report noise stays out or heavily down-ranked
- write-time provenance must support that separation
- access metrics should only reinforce what the user-facing startup payload actually returned
- managed, BYOD, and local modes must agree on those outcomes even if their internal plumbing differs
