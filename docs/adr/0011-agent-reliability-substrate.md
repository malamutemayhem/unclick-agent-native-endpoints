# ADR-0011: Agent Reliability Substrate

**Status**: Accepted
**Date**: 2026-04-30
**Supersedes/Extends**: ADR-0010 bolt-on Signals, Fishbowl check-ins, Connectors Phase 0

## Context

UnClick is moving from one-off tool calls toward a worker fleet that runs Pass products, Connectors checks, Fishbowl coordination, and later WakePass. That makes silent failure more expensive than normal request failure.

The current repo already has the first pieces:

- `api/fishbowl-watcher.ts` emits missed check-in signals and clears stale worker status.
- `api/signals-dispatch.ts` routes unread signals through user preferences.
- `mc_signals` is the user-facing alert inbox.
- Fishbowl profiles already carry `last_seen_at`, `current_status`, and `next_checkin_at`.

Those pieces should become a shared reliability substrate, not a set of product-specific rescue scripts.

## Decision

UnClick will introduce a shared agent reliability substrate before expanding Connectors, WakePass, or more XPass surfaces.

The substrate owns five primitives:

1. **Idempotent dispatch IDs**
   - Every dispatched worker task has a stable caller-provided `dispatch_id`.
   - Retrying the same dispatch returns the existing record instead of creating duplicate work.
   - The ID is generated by the caller from stable task inputs, such as source, prompt hash, target worker, and a short time bucket.

2. **Heartbeats**
   - Workers publish a compact heartbeat while working.
   - The heartbeat includes worker identity, current task, next action, ETA, blocker, and last real action time.
   - Heartbeats are metadata first. They do not carry secrets or large task payloads.

3. **Task leases**
   - A worker claims a task for a bounded lease window.
   - Work is exclusive while the lease is valid.
   - Another worker can reclaim only after expiry or explicit release.

4. **Stale reclaim**
   - A stale lease returns work to the pool.
   - Stale reclaim emits a signal and later becomes a WakePass event.
   - Reclaim is conservative. It should avoid duplicating work when a worker is only slow.

5. **Queue and duplicate telemetry**
   - The system records queue age, duplicate dispatch attempts, stale leases, and last real action.
   - These metrics decide whether the fleet is ready for more automation.

## Non-goals

This substrate is not a new Pass product.

It is not a general workflow engine.

It is not a replacement for Signals, Fishbowl, Connectors, or BackstagePass.

It does not store raw provider credentials.

It does not auto-merge code or wake humans by itself. Those are product decisions built on top.

## Product layering

| Layer | Role |
|---|---|
| Signals | User-facing inbox and notification rail |
| Fishbowl | Human and worker coordination room |
| Reliability substrate | Dispatch IDs, heartbeats, leases, stale reclaim, queue telemetry |
| Connectors | Map of connected services and route health |
| PinballWake | Reusable wake engine built on the substrate |
| WakePass | Public Pass product wrapping PinballWake for customers |

## First implementation slice

The first code slice should be deliberately small:

- Add a typed dispatch record shape.
- Add a typed heartbeat record shape.
- Add helper logic for deterministic dispatch IDs.
- Add helper logic for stale lease detection.
- Add tests for duplicate dispatch and stale reclaim decisions.

Database migration can follow as a separate PR because migrations are protected files in this repo's operating rules.

## Data model target

These are target shapes, not an instruction to create the migration in this ADR.

### `mc_agent_dispatches`

| Field | Purpose |
|---|---|
| `id` | Internal row ID |
| `api_key_hash` | Tenant boundary |
| `dispatch_id` | Caller-provided idempotency key, unique per tenant |
| `source` | Fishbowl, Connectors, WakePass, TestPass, UXPass, FlowPass, or manual |
| `target_agent_id` | Intended worker |
| `task_ref` | External task reference, if any |
| `status` | queued, leased, completed, failed, stale, cancelled |
| `lease_owner` | Worker currently holding the task |
| `lease_expires_at` | Reclaim point |
| `last_real_action_at` | Last meaningful progress |
| `created_at` | Dispatch creation time |
| `updated_at` | Last update time |
| `payload` | Minimal metadata payload |

### `mc_agent_heartbeats`

| Field | Purpose |
|---|---|
| `id` | Internal row ID |
| `api_key_hash` | Tenant boundary |
| `agent_id` | Worker identity |
| `dispatch_id` | Current dispatch, when known |
| `state` | idle, received, accepted, working, blocked, completed |
| `current_task` | Short human-readable task |
| `next_action` | What the worker intends to do next |
| `eta_minutes` | Optional ETA |
| `blocker` | Optional blocker |
| `last_real_action_at` | Last meaningful progress |
| `created_at` | Heartbeat time |

## Acceptance criteria

The first implementation is acceptable when:

- Repeated dispatch of the same `dispatch_id` returns one logical task.
- A worker can publish a heartbeat without exposing secrets.
- Stale leases can be detected from deterministic time fields.
- Stale reclaim emits or prepares a `mc_signals` event.
- Fishbowl missed check-ins can later be migrated onto the same substrate.
- WakePass can use the same records for ACK and replay instead of creating a separate wake table first.

## Consequences

Benefits:

- Connectors health checks, Fishbowl check-ins, and WakePass all share one reliability model.
- More XPass products can be launched without each inventing a private stuck-run mechanism.
- Fleet efficiency work becomes measurable instead of anecdotal.
- PinballWake starts from real operational primitives, not a brand-first scaffold.

Trade-offs:

- The first code slice is less flashy than launching WakePass immediately.
- Database work must be staged carefully because migrations and security policy files are protected.
- Product teams need to call the substrate instead of writing direct ad hoc signals for every stuck condition.
