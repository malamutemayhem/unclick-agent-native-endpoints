This is going to happen a lot. The "AI fetches our URL" design is fragile because:

1. **Egress allowlists** (this) — agents in sandboxes (Claude.ai web, some enterprise Claude Code setups, ChatGPT's browsing) won't have `unclick.world` allowlisted.

**Short answer: no hard conflicts, but a couple of things worth knowing.**

## Why there's no crash-level conflict

UnClick exposes only **9 top-level tool names** to the MCP client (4 meta-tools + 5 memory tools). The ~450 integrations (github, firecrawl, exa, gemini, etc.) live behind `unclick_call(endpoint_id, params)` — they don't pollute the tool namespace.

So even though you have standalone `github`, `firecrawl`, `exa`, `gemini`, `tavily`, `mem0`, etc. installed, none of their tool names collide with UnClick's. Claude won't get tripped up by duplicate tool names.

## Soft conflicts / redundancy

1. **Memory: mem0 vs UnClick memory** — this is the one worth thinking about. UnClick has its own built-in memory (6-layer architecture: `add_fact`, `search_memory`, etc.). If you also run mem0, Claude might write a memory to UnClick and later recall from mem0 (or vice versa) and miss it. **Pick one.** If you keep UnClick, you can disable mem0 and all your memories will live in one place.

2. **Tool overlap (github, firecrawl, exa, gemini, tavily, playwright)** — UnClick covers all of these. If you keep both connectors, Claude will sometimes call your standalone firecrawl and sometimes call UnClick's. Not broken — just redundant, and you're paying two API keys for the same capability. Safe to uninstall the standalones once you confirm UnClick's versions work for you.

3. **Old UnClick install** — if you previously installed UnClick via the stdio JSON config, **delete that entry** before adding the new remote-URL version, or you'll have two UnClick connectors competing.

## The no-worry ones

- **GitHub Integration (Anthropic's first-party)** — different thing (read-access to your repos for Claude to reference). Fine alongside UnClick.
- **Gmail, Google Calendar, Google Drive** — Anthropic's first-party Workspace connectors. Fine alongside UnClick.
- **context7, desktop-commander** — different scopes (docs, local file ops). Fine.
